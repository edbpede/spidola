## 2. Architecture overview

The system is three concentric layers. At the center sits the **Rust core**: a single Cargo workspace compiled to a static library per platform, owning every piece of logic that is not UI or playback вЂ” source ingestion, parsing, persistence, search, EPG, settings, and network fetching. Around it, the **binding layer** generated by UniFFI exposes the core's faГ§ade as idiomatic Swift (actors-friendly, async/await) and Kotlin (suspend functions, sealed classes). Outermost, two **native shells** вЂ” a SwiftUI app and a Compose-for-TV app вЂ” own everything the user sees and hears: navigation, focus, rendering, and the player engines. Playback deliberately lives in the shells, not the core: media pipelines are the most platform-entangled subsystem (surfaces, audio sessions, hardware decoders, DRM plumbing), and forcing them through a shared abstraction in Rust would invert the cost-benefit. Instead the shells implement a *shared engine contract* (В§8) specified here so both platforms behave identically at the product level.

Data flows one way per interaction: the shell calls a core service (async), the core performs I/O on its own runtime, persists to SQLite, and returns typed results or streams progress events through a callback interface; the shell renders from its own view-state, never holding business state of its own. The core is the single source of truth for all persisted data; the shells cache nothing durable except images and player-engine internals.

## 3. Modularity doctrine, repository layout, and workspace conventions

### 3.1 Modularity doctrine

The goal is **cohesion**, and the doctrine names two failure modes, not one. The first is the god-file: a unit accumulating unrelated responsibilities until no change to it is safe. The second is its inversion вЂ” confetti code: responsibilities shattered across so many fragments that understanding one behavior requires a tour of a dozen files, with the complexity merely relocated into the import graph. Rules that mechanically prevent the first reliably produce the second, so the doctrine is built on structural principles enforced by judgment, with tooling as a *signal*, never a gate.

The principles. **One unit, one reason to change:** every crate, package, module, and file owns a single concept; if its name needs "and" to describe it, it is two units. The converse holds equally вЂ” if two files cannot be understood or changed without each other, they may be one concept wrongly split. **Split at concept boundaries, on evidence:** the trigger for extracting a module is the appearance of a second nameable concept inside it (a parser growing a diagnostics ledger, a repository growing cache policy), never a size counter; a 500-line state machine that is one coherent thing beats three entangled 170-line files every time. **The newcomer test:** the working definition of "well-factored" is that a newcomer can predict from names alone where a behavior lives and what a file will contain before opening it вЂ” reviews evaluate structure against this test, not against metrics. **No junk drawers:** modules named `utils`, `helpers`, `misc`, or `manager` are banned; shared behavior earns a named home describing what it *is* (a `retry` module, a `paging` module) or it stays where it is used until a second consumer proves it shared. **Vertical slices in the apps:** UI code is organized by feature (browse, playback, sources, search, settings), never by technical kind (a global `views/`, `viewmodels/` pile), so a feature can be understood, tested, and deleted as a unit; the only horizontal layers are the design system, the core binding adapter, and a deliberately tiny platform-common layer. **Depend downward, never sideways:** features depend on core/design-system layers, never on each other; anything two features need moves down a layer explicitly. **Composition at the edge:** dependency wiring happens only in each app's thin shell and in `core-api`'s constructor path; no unit constructs its own collaborators. **Earn abstraction:** the simplest structure satisfying the requirement wins, and speculative indirection (traits with one implementation, layers with one caller) is rejected until a second concrete need exists вЂ” the engine contract (В§8) is the sanctioned exception because its second implementation ships at 1.0.

Tooling's role, precisely bounded: complexity and length lints (clippy's cognitive-complexity and length lints, detekt, SwiftLint) run at **warn with generous thresholds** and never fail CI on their own. A warning is a prompt for a one-line answer in review вЂ” "is this one concept?" вЂ” and "yes, and here's the concept" is a fully acceptable answer, recorded as a scoped, commented lint allowance rather than a global threshold bump. What reviews do reject without appeal is structural: sideways feature dependencies, junk-drawer modules, business state accumulating in shells, and units whose names have stopped describing their contents. Per-function complexity warnings are weighted more heavily than file length, because a long file of small, flat, single-purpose functions is usually healthy while a short file hiding one 200-line function usually is not.

Two structural symmetries in the tree are deliberate and load-bearing. First, the two app trees mirror each other unit for unit (CoreKit в†” corekit, PlayerContract в†” player-contract, feature slices one-to-one), which is what makes the parity policy and the FFI contract tests cheap to reason about. ### 3.3 Workspace conventions

The Rust workspace follows the project's Rust guidelines (`.augment/rules/rust-dev-pro.md`) exactly: a virtual manifest with resolver 3; `edition = "2024"` and `license = "AGPL-3.0-or-later"` set in `workspace.package`; all shared dependency versions declared once in `workspace.dependencies`; workspace-level lints (clippy pedantic at warn priority в€’1, the project's deny set for correctness lints, and the advisory complexity/length lints from В§3.1 at warn) that every member crate opts into explicitly, since lints are not inherited automatically. Modules use the modern layout вЂ” a module lives in a file of its name with children in a sibling directory; `mod.rs` files are banned as legacy style. Every crate keeps its public surface deliberate: private by default, `pub(crate)` for internal cross-module items, and a clean re-exported API from each crate root. Crate purposes, one line each: `core-model` (domain types), `core-parse` (M3U and XMLTV parsers), `core-xtream` (Xtream API client and mapping), `core-db` (SQLite persistence and migrations), `core-fetch` (HTTP), `core-search` (query layer over FTS), `core-pair` (the LAN pairing micro-server), `core-api` (the faГ§ade composing the others, carrying the UniFFI definitions вЂ” the only crate the apps ever see), and `xtask` (developer automation: binding generation, fixture refresh, release packaging вЂ” the cargo-xtask pattern rather than shell scripts).

## 4. The Rust core

### 4.1 Domain model (`core-model`)

Plain data types with serde derives where serialization is needed, following the guideline that illegal states should not construct. `Source` is an enum over its three kinds вЂ” an M3U URL variant, an M3U file variant, and an Xtream variant вЂ” each carrying only the fields that kind possesses, so an Xtream password can never exist on a file source. `Channel` carries identity, display name, group, logo reference, stream locator, media kind (live / movie / series episode), and per-channel overrides (headers, user-agent, preferred engine). `Category`, `EpgEntry`, `Favorite`, and `PlaybackHistoryEntry` complete the set. Identifiers are newtypes over integers (SourceId, ChannelId, and so on) rather than bare primitives, making cross-wiring a compile error. Stream locators are a newtype wrapping a validated URL plus optional auth material, constructed only through a parser that normalizes and validates вЂ” the "parse, don't validate" posture.

### 4.2 Parsing (`core-parse`)

The M3U parser is **streaming and incremental**: it consumes the input as chunks (from file reads or the HTTP body stream), maintains a small state machine over extinf/attribute/URL lines, and emits channels in batches through a channel-sink trait the caller provides. It never materializes the whole playlist in memory; peak parser memory is bounded by one batch regardless of playlist size, which is what makes the PRD's 50k-channel budget honest on 1 GB devices. It is tolerant by design: unknown attributes are preserved into an attribute map, malformed entries are skipped with a counted diagnostic rather than failing the import, and encodings are sniffed with a UTF-8-lossy fallback. The XMLTV parser follows the same shape (streaming pull-parsing, bounded memory, rolling-window filtering applied during parse so out-of-window programmes are never stored). Both parsers are pure вЂ” no I/O, no clocks except an injected "now" for the EPG window вЂ” which makes them trivially property-testable against the fixture corpus.

### 4.3 Xtream client (`core-xtream`)

A typed client over the Xtream Codes HTTP API: authentication handshake, category listing, live/VOD/series streams, series-info expansion, and the EPG endpoints. Responses in the wild are inconsistent (numbers as strings, missing fields, nonstandard extensions), so the deserialization layer is written defensively with custom deserializers and every mapping decision unit-tested against captured real-world fixtures (scrubbed of credentials). The client produces `core-model` types; nothing Xtream-shaped leaks upward. Stream URL construction is centralized here so credential embedding happens in exactly one audited place.

### 4.4 Persistence (`core-db`)

SQLite via **rusqlite with the bundled feature** (pinning the SQLite version identically across platforms) and rusqlite_migration for forward-only, numbered migrations applied at startup. This is a deliberate divergence from the guideline document's server-side SQL table (sqlx/sea-orm/diesel): those assume an async, networked database; an embedded, synchronous, in-process SQLite behind our own async faГ§ade is better served by rusqlite вЂ” a combination long proven in production by embedded and desktop applications alike. Connection management is a small pool (r2d2-style) with one writer and multiple readers under WAL mode. Search is **FTS5**: a contentless-delete FTS table over channel names synchronized by triggers, queried with prefix syntax for the instant-search path, with a trigram fallback ranking for fuzzy tolerance. All DB work runs on the runtime's blocking-thread facility вЂ” the guideline's rule that blocking work never sits on async worker threads is enforced structurally by making every `core-db` entry point a blocking function that only the service layer (which wraps it in the runtime's blocking adapter) may call.

Stored entities: sources (only opaque secret references for M3U/Xtream account material вЂ” see В§12), channels (with generated columns for hot filter paths), categories, favorites, playback history, EPG entries (indexed by channel and time window), settings key-values, and import diagnostics. Credential-bearing M3U channel/header/history values that must be query-adjacent are stored only as authenticated ciphertext envelopes; plaintext is recovered only by the play-time resolver. Channel refresh uses a staging-and-swap strategy inside one transaction so a failed refresh can never leave a source half-updated, and favorites/hidden flags survive refresh by keying on a stable per-source channel identity rather than row ids. Actual M3U imports derive that identity with a catalog-keyed HMAC (over `tvg-id`, or a length-prefixed URL/name fallback), so a stored identity cannot act as an offline verifier for a credential-bearing locator; deterministic public hashes remain only in non-secret fixture/tooling paths.

### 4.5 Fetching (`core-fetch`)

All HTTP lives here: reqwest configured with **rustls** (no OpenSSL anywhere in the dependency tree вЂ” this is what keeps Android and Apple cross-compilation sane), sensible timeouts (connect, read, and an overall import deadline), redirect policy with a hop cap, per-source custom user-agent and header injection, and body **streaming** handed directly to the parser sink so playlist bytes flow network в†’ parser в†’ DB batch without full buffering. Certificate handling follows platform trust; an opt-in per-source "accept invalid TLS" escape hatch exists because self-hosted headends with self-signed certs are common вЂ” it is off by default, loudly labeled, and scoped to that source only.

### 4.6 Services and the faГ§ade (`core-api`)

`core-api` composes the crates into the surface the apps consume. It owns a **Tokio multi-thread runtime** created once at library initialization (the shells never see it), and exposes services as UniFFI interfaces: a SourceService (add, list, refresh with progress, remove, disable), a CatalogService (browse queries, paged), a SearchService (query in, ranked page out), a FavoritesService, an EpgService, a SettingsService, and a PairingService (start/stop the LAN server, report its URL, surface submissions as events). Long operations (import, refresh, EPG ingest) follow one pattern: the call returns quickly with a task handle; progress, completion, and failure arrive through a registered listener callback; cancellation is explicit via the handle and is honest (checked at batch boundaries, guaranteed to stop within one batch). Read paths for the UI are **paged by contract** вЂ” every list-returning method takes an offset/limit or cursor вЂ” so no FFI call ever transfers an unbounded collection.

### 4.7 Error handling вЂ” the cross-layer policy

Errors are designed once, end to end, and every layer follows its language's guidelines. **In the core**, per the Rust guidelines: every core crate defines its own precise error enum with thiserror v2, wired with source chains; `anyhow` appears only in `xtask` and test code, never in library crates; panics are reserved for programming errors and are treated as bugs (a panic crossing the FFI is a release blocker, caught in contract tests). **At the FFI boundary**, `core-api` flattens the internal taxonomy into a small, stable, user-mappable error enum вЂ” the shells receive variants like NetworkUnreachable, Unauthorized, ParseFailed-with-diagnostics, StorageCorrupt, Cancelled вЂ” each carrying the minimal structured data the UI needs to render the PRD's actionable error requirement. **In the Swift shell**, per the Swift guidelines: the CoreKit adapter maps the boundary enum onto a typed Swift error enum conforming to the standard error protocol, thrown through async functions and handled with exhaustive switches; no stringly-typed or untyped error paths reach feature code. **In the Kotlin shell**, per the Kotlin guidelines: expected, domain-meaningful failures are modeled as sealed hierarchies the compiler forces callers to handle exhaustively in `when`; exceptions are reserved for programming errors and violated invariants (the precondition functions used for fail-fast intent); `Result`/`runCatching` is confined to wrapping throwing calls locally and never leaks across module boundaries as a public contract; and cancellation exceptions are always rethrown, never swallowed, so structured concurrency stays intact. **In both shells**, every error the user can see maps to one of the PRD's plain-language failure classes with its prescribed actions (retry, try the other player, go back) вЂ” an error with no action is a design bug. Internal detail (the full chain) is preserved into the log stream, not the FFI error, so user-facing errors stay clean while diagnostics stay rich.

### 4.8 Observability вЂ” the cross-layer logging policy

Logging is one pipeline with three sinks. The core uses `tracing` with spans around imports, refreshes, and queries, and structured fields rather than formatted strings. A small FFI callback interface lets each shell install itself as the subscriber sink. On tvOS, per the Swift guidelines, the sink forwards into **OSLog's Logger** вЂ” never `print` for diagnostics вЂ” with the app bundle identifier as the subsystem, one category per core span target (import, db, fetch, playback), levels mapped one-to-one, and privacy redaction annotations applied so free-form values default private. On Android, the sink forwards to **logcat** with structured, per-target tags and matching level mapping; shell-side code logs through the same tag scheme so a single logcat filter shows the whole pipeline. Log level is a runtime setting (PRD diagnostics screen), applied at the tracing filter so disabled levels cost nothing. Player engines are wired into the same scheme: engine state transitions and EngineError emissions are logged at info/error with the channel's non-identifying id, giving support threads a coherent story across core and engine. The log-export action serializes the core's ring buffer of recent events to a shareable text file, scrubbing anything credential-shaped by construction (secrets never enter log messages because the secret types' debug representations are redacted вЂ” enforced by a custom Debug impl and a clippy-level CI grep for accidental formatting of those types; the same rule applies shell-side, where secret values never appear in interpolated log messages).

## 5. FFI boundary (UniFFI)

UniFFI in **proc-macro mode** (attributes on the Rust definitions; no separate UDL files to drift). The generated Swift package and Kotlin library are build artifacts, produced by `xtask` and consumed by the app builds вЂ” never hand-edited, never committed except as versioned release artifacts for reproducibility checks. Boundary design rules, each load-bearing: records crossing the boundary are flat, owned data (no borrowed views); every service method that touches I/O is declared async, surfacing natively as Swift async functions and Kotlin suspend functions on the core's runtime; event push uses UniFFI callback interfaces with the explicit contract that callbacks may arrive on any thread and the shells must trampoline to their main actors/dispatchers themselves; enums crossing the boundary are exhaustively matched on the shell side with an "unknown future variant" arm reserved so the core can grow without breaking older shells during development; and the boundary is versioned вЂ” a single handshake method reports core, schema, and boundary versions, checked at app startup so a mismatched framework fails fast and legibly instead of corrupting data. The current handshake is schema **2** / boundary **4**. M3U channel locators and credential-bearing header values cross this boundary only as opaque authenticated envelopes; both shells must call the core resolver immediately before engine construction. The resolver returns opaque UniFFI objects rather than generated records, so Swift reflection and Kotlin data-class diagnostics cannot enumerate the ephemeral locator, user-agent, or header values; explicit getters are the only native access path.

The threading contract in one sentence: the shells own their UI threads, the core owns all its threads, and the only crossings are async returns (delivered by the generated bindings onto the caller's continuation) and callbacks (delivered on core threads, trampolined by shell code).

## 8. The player engine contract

Both shells implement the same conceptual interface, specified here so product behavior is identical. An engine exposes: a load operation taking a stream locator, header map, user-agent, and a buffering profile; transport operations (play, pause, stop, seek-if-seekable); track enumeration and selection for audio and subtitles; aspect-ratio mode cycling; and a read-only state stream emitting a small state machine вЂ” idle, loading, buffering, playing, paused, ended, failed-with-EngineError. EngineError is a shared taxonomy with exactly the classes the PRD's error UX needs: SourceUnreachable, Unauthorized, UnsupportedFormat, DecoderFailed, Timeout, and Unknown-with-detail. Engines must be disposable and re-creatable cheaply, because zapping destroys and rebuilds them constantly; the channel strip's zap path is the performance-critical consumer and its budget (video visible within the PRD's two-second bar, UI never blocked) is an acceptance test per engine.

Engine selection policy, identical on both platforms: resolve per-channel override, else per-source override, else platform default (ExoPlayer on Android, MPVKit on tvOS). **Automatic fallback** triggers only on UnsupportedFormat or DecoderFailed from the default engine: the shell offers a one-button "Try other player" (and a remember-for-this-channel toggle), rather than silently switching вЂ” silent switching would make engine bugs invisible and support impossible. All engine parameters the settings screen exposes (buffering profile, subtitle styling) map through the contract so settings language stays engine-neutral.

Toolchains are pinned: Rust 1.96.1 via rust-toolchain file; Xcode and Android SDK/NDK versions recorded in `docs/toolchains` and asserted by build scripts. ## 10. Testing strategy

The pyramid is bottom-heavy in Rust, where the logic is. Parsers get property-based tests (random mutation of fixture playlists must never panic and must preserve the skipped-entry accounting invariant) plus a golden-fixture corpus grown from every real-world playlist bug ever reported вЂ” the corpus is the project's crown jewel and lives in `fixtures/` with provenance notes. `core-db` gets migration tests (every historical schema upgraded to head and smoke-queried) and the staging-swap refresh property (a refresh failing at any injected point leaves the prior catalog intact). The Xtream client tests against captured, scrubbed response fixtures with a local stub server. The FFI boundary gets **contract tests**: a small suite executed from both Swift and Kotlin against the real compiled core, asserting the same fixture yields the same results through both bindings вЂ” this is what keeps "parity" from being a slogan. Shell-side, view models are unit-tested against a fake CoreKit; each platform keeps a thin UI smoke suite (launch, add fixture source, reach playback screen with a stub engine) rather than a brittle full E2E net. Engine implementations get device-farm spot checks against a maintainer-operated test headend serving self-produced streams in the formats the taxonomy names вЂ” this is manual-with-checklist per release, honestly acknowledged as the least automatable layer.

## 11. Performance engineering

The PRD budgets are enforced as follows: import throughput and parser memory ceilings are asserted by criterion benchmarks in the core lane with regression gates; search latency is benchmarked at a generated 50k-channel dataset; the shells apply platform tooling per release checklist (Instruments on tvOS, Macrobenchmark and Perfetto on Android) with the low-end Chromecast-class device as the Android baseline. Structural rules that make the budgets reachable rather than aspirational: no synchronous core call on a UI thread (the FFI's async-only surface makes this nearly unrepresentable), all lists virtualized, image decode off-main, DB access through indexed query plans reviewed when query shapes change (an xtask target dumps EXPLAIN output for the hot queries), and channel-zap treated as the sacred path вЂ” profiled every release on both platforms.

## 12. Security, privacy, and license engineering

Plaintext credentials never persist in SQLite or logs. The core defines a host-secrets callback interface вЂ” store, retrieve, delete by opaque key вЂ” implemented on tvOS over Keychain and on Android over Keystore-backed encrypted preferences. Xtream credentials and M3U source URLs/user agents live there directly, with SQLite storing only opaque references. Imported M3U channel locators, per-channel user agents/header values, and playback-history locators can number in the tens of thousands and must remain query-adjacent, so they use an equivalent encrypted-at-rest design: ChaCha20-Poly1305 authenticated envelopes in SQLite, keyed by one random catalog key held only in the platform secure store. Locator and header-value envelopes use separate authenticated domains and M3U resolution requires a valid envelope, so prefix removal and cross-field ciphertext substitution fail closed. Envelopes remain opaque through navigation and FFI and are opened only by `SourceService::resolve_playback` immediately before playback, returning opaque ephemeral locator/header objects whose Rust Debug, Swift reflection, Kotlin `toString()`, and shell request diagnostics all redact plaintext. Secret-store writes and deletes fail closed; a source is never committed when its secure material was not durably stored. Secret-bearing Rust values redact Debug output, zeroize on drop where owned as secret types, and never implement serde for raw values. Schema 2 is the pre-1.0 security cutover: it discards legacy M3U URL/file rows that could contain plaintext, then a durable `pending`/`complete` marker makes checkpoint-and-vacuum retry on every open until the forensic scrub succeeds; raw database/WAL and log-stream regressions pin that guarantee.

The pairing server is reachable only from the local network, exists only while its screen is visible, serves a static form and accepts one POST shape, and displays a session-random token in the TV UI that must accompany submissions вЂ” a person on the network cannot inject a source into a TV they cannot see. Locality is enforced as a **peer check on each connection** (private, link-local, or loopback, refused before a byte of the request is read), not as a selective bind: binding to "only the LAN interfaces" would require enumerating them, which the core provably cannot do (В§14, "The shells resolve the pairing server's advertised address"). The peer check is the stronger property anyway вЂ” it judges the connection actually in front of the server rather than assuming an interface's reachability вЂ” and it is what the token, the screen-bound lifetime, and the address validation all layer on top of. TLS is rustls everywhere with platform roots; the self-signed escape hatch is per-source and surfaced in the source list as a persistent badge.

License engineering: every file carries an SPDX header (AGPL-3.0-or-later for project code) checked by a REUSE lint in CI; cargo-deny enforces a dependency license allow-list (permissive plus LGPL, with copyleft-incompatible licenses denied) **over the Rust graph only** вЂ” cargo-deny audits Cargo's dependency tree and is structurally blind to the shells', so the JVM/Gradle graph (Media3, Compose, Hilt, JNA, zxing) and the SPM graph (MPVKit) have no automated license gate today and are checked by hand at review. That is a real hole in "all bundled components must be AGPL-compatible" (PRD В§10) and it is recorded rather than papered over: the fix is an allow-list-or-fail gate per graph вЂ” `app.cash.licensee` is the Gradle analogue of cargo-deny вЂ” and it is scoped as a Phase 7 release-engineering task, since that phase already owns the final license audit. Until it lands, a new shell dependency's license is a reviewer's job, and running `cargo deny check` after adding one proves nothing about it; the mpv/FFmpeg builds on both platforms are pinned to LGPL configurations with their build flags committed вЂ” resolving to LGPL-3.0 on both platforms, forced by the Apache-2.0 TLS providers and GMP and accepted in PRD В§10 (mpv itself remains LGPL-2.1-or-later) вЂ” satisfying both AGPL compatibility and the App Store posture discussed in the PRD; and every screen the pairing server renders includes the source-code link that keeps AGPL section 13 compliance trivially true. Third-party notices are generated at build time into the About screen from the dependency graph, not maintained by hand.

## 13. Versioning and release

Semantic versioning on the app; the core carries an independent schema version (DB) and boundary version (FFI) checked in the startup handshake. This revision is schema **2** / boundary **4**. Both shells fail before bootstrap when either value differs from what they support. DB migrations are forward-only; a downgraded app refusing a newer schema fails with a clear message rather than guessing.

**Until 1.0 ships, no compatibility is owed to existing data (decided).** The app is unreleased: there are no users, and every database in existence holds a fixture catalog someone can re-import. So a change that orphans stored data вЂ” a renamed settings key, a reshaped row вЂ” is accepted rather than migrated, and no effort is spent on shims that would exist only to preserve a developer's fixture. This is a rule about *data*, not about the machinery: schema changes still land as numbered forward-only migrations (В§4.4), the handshake still refuses a newer schema legibly, and the boundary version still moves when the exported surface changes shape вЂ” those exist to make the *next* change safe, and Phase 7's exit criteria test them against a deliberately stale shell. The rule expires at 1.0, when the first real database stops being disposable. Release cadence follows the PRD milestones, with each release documented by a changelog generated from conventional commits. Architecture-level decisions made or changed after this document are captured by amending this document by PR like any code.

## 14. Appendix вЂ” decision log

**Rust core with native shells over any cross-platform UI framework.** Flutter lacks tvOS support entirely and is second-class on Android TV; Kotlin Multiplatform would share logic but forfeit the Rust engine the project explicitly wants. UniFFI gives both platforms idiomatic bindings from one definition. Consequence accepted: two UIs to maintain вЂ” mitigated by the shared engine contract and FFI contract tests.

**rusqlite over sqlx/diesel.** Embedded, in-process, single-writer SQLite behind our own async faГ§ade; the async-SQL ecosystem's costs buy nothing here, and bundling pins one SQLite version across both platforms. Consequence accepted: hand-written SQL вЂ” mitigated by migration tests and the EXPLAIN xtask.

**ExoPlayer default on Android, MPVKit default on tvOS.** On Android, Media3 is the platform-integrated path (media session, voice transport, hardware decode) and handles mainstream IPTV formats; mpv is the codec-breadth fallback. On tvOS, AVPlayer's format range is too narrow to be the default for IPTV reality, so the positions invert. Consequence accepted: different default behavior per platform вЂ” mitigated by the identical selection/fallback policy and shared error taxonomy.

**Loud fallback over silent engine switching.** Silent switching hides engine regressions and produces unexplainable behavior differences between users. The one-button fallback with remember-toggle keeps agency with the user and signal with the maintainer.

**Playback in the shells, not the core.** Media pipelines are the most platform-bound subsystem; abstracting them in Rust would maximize, not minimize, per-platform glue. The engine contract captures the sharable part вЂ” semantics вЂ” without sharing the unsharable part вЂ” implementation.

**AGPL-3.0-or-later with LGPL media libraries.** Project code is AGPL-3.0-or-later (decided) to guarantee derivative freedom including network use with future-proofing; embedded FFmpeg/mpv are built LGPL-only for App Store viability and clean compatibility. The stack's effective version is LGPL-3.0 on both platforms вЂ” every TLS route forces it (the Apache-2.0 providers require `--enable-version3`; GnuTLS drags in LGPL-3.0 GMP) вЂ” accepted in PRD В§10 together with the LGPLv3-specific App Store consideration it raises. The contributor-agreement model is decided: DCO plus the App Store Distribution Exception (`APPSTORE_EXCEPTION.md`), recorded in `CONTRIBUTING.md` and PRD В§10 вЂ” no CLA, no copyright assignment.

**Modularity doctrine: cohesion by principle, tooling as signal (В§3.1).** An earlier draft enforced a hard ~300-line file ceiling in CI; it was rejected because mechanical size gates optimize the metric rather than the goal, pushing splits to wherever the linter fires instead of concept boundaries and trading the god-file failure mode for its confetti-code inversion. The adopted doctrine enforces structural rules absolutely (single-concept units, no junk drawers, vertical slices, downward-only dependencies, edge-only composition, earned abstraction) while running complexity/length lints at warn as review prompts with scoped, justified allowances. Consequence accepted: the standard rests on review judgment rather than automation вЂ” judged correct because cohesion is not machine-checkable, and mitigated by the newcomer test giving reviews a concrete, arguable criterion.

**Xtream buffers each listing where M3U streams (Phase 6).** В§4.2's streaming, memory-bounded posture is a property of the *input format*, not a universal one, and Xtream cannot honour it: a catalog arrives as one JSON array per listing with no pagination and no envelope to resume from, so the body must be whole before it can be read. The alternative вЂ” fetching per category to bound each response вЂ” was rejected for now because it trades one large response for N round trips on the import path without changing the worst case (a headend may put every channel in one category). Consequence accepted: peak import memory for an Xtream source is one listing rather than one batch, bounded by a 64 MiB cap in `core-xtream/src/request.rs` against a ~20 MB worst honest case вЂ” the one place in the core where the 1 GB-device budget rests on a ceiling rather than on structure. Mitigated by the cap being a sink-side refusal (an oversized body is dropped as it arrives, never buffered then rejected), and flagged for measurement against the low-end baseline in Phase 7; per-category fetching is the escape hatch if it does not hold.

A second, larger growth sits behind the same flag and is *not* covered by that cap. Series are two-level вЂ” list the shows, then one info call per show вЂ” and `core-api/src/xtream.rs` accumulates every expanded episode into one `collected.channels` vector before any of it is staged. The 64 MiB ceiling bounds each individual response, so it never sees this: the peak here is the whole episode set of the whole catalog, summed across hundreds of responses that were each perfectly small. For a series-heavy headend that is the import's real high-water mark, and "one listing" understates it. The shape of the fix is known and unglamorous вЂ” stage in batches at the show boundary, which the staging path already supports for M3U вЂ” and is deliberately not taken here: the Phase 7 low-end-baseline measurement must cover the series path specifically and size the batch against a real device, rather than have a number guessed at now. Consequence accepted: until that measurement, a series-heavy Xtream import is the one path whose peak memory is bounded by neither structure nor ceiling.

**Xtream catalogs persist a credential-free locator (Phase 6).** Xtream is the one protocol Spidola speaks that puts the account password *in the stream URL*, which would have put a credential in SQLite the moment a catalog was stored вЂ” contradicting В§12. The catalog therefore stores a locator with no credential in it, and `core-xtream/src/urls.rs` вЂ” the single audited embedding point вЂ” resolves it to a playable URL at play time. Consequence accepted: playback cannot use a stored locator directly for Xtream sources and must resolve through the core, which is the same shape В§4.5 already requires for authed artwork. Judged worth it because it keeps plaintext credentials out of SQLite by construction rather than by discipline, and is proven by a contract test that searches the raw bytes of every SQLite file (including the WAL) for the password after a real account is added.

**M3U credential material uses authenticated encryption at rest (pre-Phase-7 validation).** Unlike Xtream, an M3U provider can put account material anywhere: the playlist URL, source user agent, every channel locator, VLC user-agent/referrer directives, and then playback history. Keeping one secure-store entry per imported field would turn a 50k-channel catalog into tens of thousands of Keychain/Keystore operations and make refresh staging non-transactional. The chosen hybrid keeps low-cardinality source values directly in the host secure store and stores high-cardinality catalog/history values as ChaCha20-Poly1305 envelopes under one random catalog key that never leaves that store. The envelopes are intentionally opaque FFI/navigation values and are idempotently sealed so recents cannot double-wrap them; strict, domain-separated play-time resolution opens the locator and request overrides only. The same catalog key derives stable HMAC channel identities, including the URL/name fallback, so SQLite does not retain a public hash that can confirm guesses about a credential URL. A plaintext-template design was rejected because M3U has no reliable credential grammar, and field-by-field secure-store entries were rejected for the scale/transactionality cost above. Consequence accepted: losing the catalog key makes encrypted catalog values unreadable and requires re-import, which is the correct pre-1.0 failure mode; ciphertext modification, prefix removal, and cross-field substitution are rejected rather than producing a guessed request. Schema 2 deletes legacy M3U rows because compatibility is not owed before 1.0; a durable migration marker retries checkpoint/vacuum after crashes, and raw SQLite/WAL/log tests prove the cutover instead of relying on column inspection.

**Stream Red's lightness is a contrast floor, not a taste call (Phase 6).** The accessibility pass audited every foreground/background pair the shells actually use and found the palette passing WCAG AA everywhere except one: Stream Red, the semantic error color, was `#C0554E` and reached only 4.05:1 on Studio and 3.58:1 on Set against the 4.5:1 floor for body text. It is the one semantic color that carries prose вЂ” a validation message on the add-source screen in both shells, and Material's `error` role on Android вЂ” so the icon-only 3:1 allowance did not cover it. PRD В§8.2 pins hexes for its five named values and describes semantic red only qualitatively ("muted to sit in the same tonal family"), so the hex was ours to choose and this is not a PRD amendment. Stream Red is now `#C96E69`: the same hue (4В°) and saturation (48%), lightness raised 53% в†’ 60%, which reaches 4.56:1 on Set and 5.16:1 on Studio вЂ” still muted, still in the family, and now legible as text. **The shells own the user-facing error vocabulary; the core returns a code and structured fields (Phase 6, implementation deferred).** `ActionableError` currently crosses the FFI as finished English вЂ” a `failureClass`, a `message`, and `ErrorAction` labels ("Try again", "Go back", "Edit") вЂ” which made it the one user-facing surface the localization pass could not touch. Words have to live where the translation infrastructure lives, and that is the shells: `.xcstrings` on tvOS, `strings.xml` on Android, each with a platform's plural rules, its gender and case agreement, and its reviewers. A core that ships English has no way to grow a second language, and a core that ships every language is a translation platform written in Rust. The decision is therefore that the core stops choosing words: it returns a stable code plus the data the sentence needs (the host that failed, the seconds remaining), and each shell renders that into its own catalog. The alternative вЂ” passing the shell's locale down and localizing core-side вЂ” was rejected because it would put a catalog behind the FFI and still deny the shells the case and agreement rules they need at the call site. Consequence accepted: the implementation is not in this phase. It is a В§5 boundary change (a code taxonomy, both shells, six bridge files, a version bump), and bundling it into the phase that discovered it would have meant a boundary change reviewed as a footnote to a localization sweep. Mitigated by leaving the *entire* `ActionableError` surface unextracted rather than half-extracted вЂ” the one place in either shell where visible English is deliberate, so it reads as pending work rather than as an oversight, and cannot be mistaken for done.

**The shells resolve the pairing server's advertised address (Phase 6).** В§2 gives the core all non-UI logic, and discovering the TV's LAN address looks like exactly that вЂ” but the core provably cannot do it. `core-pair` infers the address by asking the kernel which interface leaves the host, which is correct on a plain LAN and wrong on any multi-homed host: measured on a VPN box, every probe destination (public, private, broadcast, multicast) returned the tunnel address, and only a destination already inside the target subnet returned the LAN address вЂ” which is the fact being sought. Adding `if-addrs` for real interface enumeration was considered and deferred: it adds a workspace dependency and rests on `getifaddrs` availability at the Android minimum SDK, to solve a problem the shells do not have (`NWInterface` on tvOS, `WifiManager`/`NetworkInterface` on Android). Consequence accepted: one sanctioned piece of non-UI logic lives in each shell, and `PairingService::start` takes the address as an argument with core inference as a convenience path. Mitigated by the invariant that the server advertises only addresses it would answer on (the advertised host is checked with the same predicate as the peer check), and by security not depending on it at all вЂ” locality is enforced by the peer check on the connection, never by the URL.

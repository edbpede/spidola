## 2. Architecture overview

The system is three concentric layers. At the center sits the **Rust core**: a single Cargo workspace compiled to a static library per platform, owning every piece of logic that is not UI or playback — source ingestion, parsing, persistence, search, EPG, settings, and network fetching. Around it, the **binding layer** generated by UniFFI exposes the core's façade as idiomatic Swift (actors-friendly, async/await) and Kotlin (suspend functions, sealed classes). Outermost, two **native shells** — a SwiftUI app and a Compose-for-TV app — own everything the user sees and hears: navigation, focus, rendering, and the player engines. Playback deliberately lives in the shells, not the core: media pipelines are the most platform-entangled subsystem (surfaces, audio sessions, hardware decoders, DRM plumbing), and forcing them through a shared abstraction in Rust would invert the cost-benefit. Instead the shells implement a *shared engine contract* (§8) specified here so both platforms behave identically at the product level.

Data flows one way per interaction: the shell calls a core service (async), the core performs I/O on its own runtime, persists to SQLite, and returns typed results or streams progress events through a callback interface; the shell renders from its own view-state, never holding business state of its own. The core is the single source of truth for all persisted data; the shells cache nothing durable except images and player-engine internals.

## 3. Modularity doctrine, repository layout, and workspace conventions

### 3.1 Modularity doctrine

The goal is **cohesion**, and the doctrine names two failure modes, not one. The first is the god-file: a unit accumulating unrelated responsibilities until no change to it is safe. The second is its inversion — confetti code: responsibilities shattered across so many fragments that understanding one behavior requires a tour of a dozen files, with the complexity merely relocated into the import graph. Rules that mechanically prevent the first reliably produce the second, so the doctrine is built on structural principles enforced by judgment, with tooling as a *signal*, never a gate.

The principles. **One unit, one reason to change:** every crate, package, module, and file owns a single concept; if its name needs "and" to describe it, it is two units. The converse holds equally — if two files cannot be understood or changed without each other, they may be one concept wrongly split. **Split at concept boundaries, on evidence:** the trigger for extracting a module is the appearance of a second nameable concept inside it (a parser growing a diagnostics ledger, a repository growing cache policy), never a size counter; a 500-line state machine that is one coherent thing beats three entangled 170-line files every time. **The newcomer test:** the working definition of "well-factored" is that a newcomer can predict from names alone where a behavior lives and what a file will contain before opening it — reviews evaluate structure against this test, not against metrics. **No junk drawers:** modules named `utils`, `helpers`, `misc`, or `manager` are banned; shared behavior earns a named home describing what it *is* (a `retry` module, a `paging` module) or it stays where it is used until a second consumer proves it shared. **Vertical slices in the apps:** UI code is organized by feature (browse, playback, sources, search, settings), never by technical kind (a global `views/`, `viewmodels/` pile), so a feature can be understood, tested, and deleted as a unit; the only horizontal layers are the design system, the core binding adapter, and a deliberately tiny platform-common layer. **Depend downward, never sideways:** features depend on core/design-system layers, never on each other; anything two features need moves down a layer explicitly. **Composition at the edge:** dependency wiring happens only in each app's thin shell and in `core-api`'s constructor path; no unit constructs its own collaborators. **Earn abstraction:** the simplest structure satisfying the requirement wins, and speculative indirection (traits with one implementation, layers with one caller) is rejected until a second concrete need exists — the engine contract (§8) is the sanctioned exception because its second implementation ships at 1.0.

Tooling's role, precisely bounded: complexity and length lints (clippy's cognitive-complexity and length lints, detekt, SwiftLint) run at **warn with generous thresholds** and never fail CI on their own. A warning is a prompt for a one-line answer in review — "is this one concept?" — and "yes, and here's the concept" is a fully acceptable answer, recorded as a scoped, commented lint allowance rather than a global threshold bump. What reviews do reject without appeal is structural: sideways feature dependencies, junk-drawer modules, business state accumulating in shells, and units whose names have stopped describing their contents. Per-function complexity warnings are weighted more heavily than file length, because a long file of small, flat, single-purpose functions is usually healthy while a short file hiding one 200-line function usually is not.

### 3.3 Workspace conventions

The Rust workspace follows the project's Rust guidelines (`.augment/rules/rust-dev-pro.md`) exactly: a virtual manifest with resolver 3; `edition = "2024"` and `license = "AGPL-3.0-or-later"` set in `workspace.package`; all shared dependency versions declared once in `workspace.dependencies`; workspace-level lints (clippy pedantic at warn priority −1, the project's deny set for correctness lints, and the advisory complexity/length lints from §3.1 at warn) that every member crate opts into explicitly, since lints are not inherited automatically. Modules use the modern layout — a module lives in a file of its name with children in a sibling directory; `mod.rs` files are banned as legacy style. Every crate keeps its public surface deliberate: private by default, `pub(crate)` for internal cross-module items, and a clean re-exported API from each crate root. Crate purposes, one line each: `core-model` (domain types), `core-parse` (M3U and XMLTV parsers), `core-xtream` (Xtream API client and mapping), `core-db` (SQLite persistence and migrations), `core-fetch` (HTTP), `core-search` (query layer over FTS), `core-pair` (the LAN pairing micro-server), `core-api` (the façade composing the others, carrying the UniFFI definitions — the only crate the apps ever see), and `xtask` (developer automation: binding generation, fixture refresh, release packaging — the cargo-xtask pattern rather than shell scripts).

## 4. The Rust core

### 4.1 Domain model (`core-model`)

Plain data types with serde derives where serialization is needed, following the guideline that illegal states should not construct. `Source` is an enum over its three kinds — an M3U URL variant, an M3U file variant, and an Xtream variant — each carrying only the fields that kind possesses, so an Xtream password can never exist on a file source. `Channel` carries identity, display name, group, logo reference, stream locator, media kind (live / movie / series episode), and per-channel overrides (headers, user-agent, preferred engine). `Category`, `EpgEntry`, `Favorite`, and `PlaybackHistoryEntry` complete the set. Identifiers are newtypes over integers (SourceId, ChannelId, and so on) rather than bare primitives, making cross-wiring a compile error. Stream locators are a newtype wrapping a validated URL plus optional auth material, constructed only through a parser that normalizes and validates — the "parse, don't validate" posture.

### 4.2 Parsing (`core-parse`)

The M3U parser is **streaming and incremental**: it consumes the input as chunks (from file reads or the HTTP body stream), maintains a small state machine over extinf/attribute/URL lines, and emits channels in batches through a channel-sink trait the caller provides. It never materializes the whole playlist in memory; peak parser memory is bounded by one batch regardless of playlist size, which is what makes the PRD's 50k-channel budget honest on 1 GB devices. It is tolerant by design: unknown attributes are preserved into an attribute map, malformed entries are skipped with a counted diagnostic rather than failing the import, and encodings are sniffed with a UTF-8-lossy fallback. The XMLTV parser follows the same shape (streaming pull-parsing, bounded memory, rolling-window filtering applied during parse so out-of-window programmes are never stored). Both parsers are pure — no I/O, no clocks except an injected "now" for the EPG window — which makes them trivially property-testable against the fixture corpus.

### 4.3 Xtream client (`core-xtream`)

A typed client over the Xtream Codes HTTP API: authentication handshake, category listing, live/VOD/series streams, series-info expansion, and the EPG endpoints. Responses in the wild are inconsistent (numbers as strings, missing fields, nonstandard extensions), so the deserialization layer is written defensively with custom deserializers and every mapping decision unit-tested against captured real-world fixtures (scrubbed of credentials). The client produces `core-model` types; nothing Xtream-shaped leaks upward. Stream URL construction is centralized here so credential embedding happens in exactly one audited place.

### 4.4 Persistence (`core-db`)

SQLite via **rusqlite with the bundled feature** (pinning the SQLite version identically across platforms) and rusqlite_migration for forward-only, numbered migrations applied at startup. This is a deliberate divergence from the guideline document's server-side SQL table (sqlx/sea-orm/diesel): those assume an async, networked database; an embedded, synchronous, in-process SQLite behind our own async façade is better served by rusqlite — a combination long proven in production by embedded and desktop applications alike. Connection management is a small pool (r2d2-style) with one writer and multiple readers under WAL mode. Search is **FTS5**: a contentless-delete FTS table over channel names synchronized by triggers, queried with prefix syntax for the instant-search path, with a trigram fallback ranking for fuzzy tolerance. All DB work runs on the runtime's blocking-thread facility — the guideline's rule that blocking work never sits on async worker threads is enforced structurally by making every `core-db` entry point a blocking function that only the service layer (which wraps it in the runtime's blocking adapter) may call.

Stored entities: sources (sans secrets — see §12), channels (with generated columns for hot filter paths), categories, favorites, playback history, EPG entries (indexed by channel and time window), settings key-values, and import diagnostics. Channel refresh uses a staging-and-swap strategy inside one transaction so a failed refresh can never leave a source half-updated, and favorites/hidden flags survive refresh by keying on a stable per-source channel identity hash rather than row ids.

### 4.5 Fetching (`core-fetch`)

All HTTP lives here: reqwest configured with **rustls** (no OpenSSL anywhere in the dependency tree — this is what keeps Android and Apple cross-compilation sane), sensible timeouts (connect, read, and an overall import deadline), redirect policy with a hop cap, per-source custom user-agent and header injection, and body **streaming** handed directly to the parser sink so playlist bytes flow network → parser → DB batch without full buffering. Certificate handling follows platform trust; an opt-in per-source "accept invalid TLS" escape hatch exists because self-hosted headends with self-signed certs are common — it is off by default, loudly labeled, and scoped to that source only.

### 4.6 Services and the façade (`core-api`)

`core-api` composes the crates into the surface the apps consume. It owns a **Tokio multi-thread runtime** created once at library initialization (the shells never see it), and exposes services as UniFFI interfaces: a SourceService (add, list, refresh with progress, remove, disable), a CatalogService (browse queries, paged), a SearchService (query in, ranked page out), a FavoritesService, an EpgService, a SettingsService, and a PairingService (start/stop the LAN server, report its URL, surface submissions as events). Long operations (import, refresh, EPG ingest) follow one pattern: the call returns quickly with a task handle; progress, completion, and failure arrive through a registered listener callback; cancellation is explicit via the handle and is honest (checked at batch boundaries, guaranteed to stop within one batch). Read paths for the UI are **paged by contract** — every list-returning method takes an offset/limit or cursor — so no FFI call ever transfers an unbounded collection.

### 4.7 Error handling — the cross-layer policy

Errors are designed once, end to end, and every layer follows its language's guidelines. **In the core**, per the Rust guidelines: every core crate defines its own precise error enum with thiserror v2, wired with source chains; `anyhow` appears only in `xtask` and test code, never in library crates; panics are reserved for programming errors and are treated as bugs (a panic crossing the FFI is a release blocker, caught in contract tests). **At the FFI boundary**, `core-api` flattens the internal taxonomy into a small, stable, user-mappable error enum — the shells receive variants like NetworkUnreachable, Unauthorized, ParseFailed-with-diagnostics, StorageCorrupt, Cancelled — each carrying the minimal structured data the UI needs to render the PRD's actionable error requirement. **In the Swift shell**, per the Swift guidelines: the CoreKit adapter maps the boundary enum onto a typed Swift error enum conforming to the standard error protocol, thrown through async functions and handled with exhaustive switches; no stringly-typed or untyped error paths reach feature code. **In the Kotlin shell**, per the Kotlin guidelines: expected, domain-meaningful failures are modeled as sealed hierarchies the compiler forces callers to handle exhaustively in `when`; exceptions are reserved for programming errors and violated invariants (the precondition functions used for fail-fast intent); `Result`/`runCatching` is confined to wrapping throwing calls locally and never leaks across module boundaries as a public contract; and cancellation exceptions are always rethrown, never swallowed, so structured concurrency stays intact. **In both shells**, every error the user can see maps to one of the PRD's plain-language failure classes with its prescribed actions (retry, try the other player, go back) — an error with no action is a design bug. Internal detail (the full chain) is preserved into the log stream, not the FFI error, so user-facing errors stay clean while diagnostics stay rich.

### 4.8 Observability — the cross-layer logging policy

Logging is one pipeline with three sinks. The core uses `tracing` with spans around imports, refreshes, and queries, and structured fields rather than formatted strings. A small FFI callback interface lets each shell install itself as the subscriber sink. On tvOS, per the Swift guidelines, the sink forwards into **OSLog's Logger** — never `print` for diagnostics — with the app bundle identifier as the subsystem, one category per core span target (import, db, fetch, playback), levels mapped one-to-one, and privacy redaction annotations applied so free-form values default private. On Android, the sink forwards to **logcat** with structured, per-target tags and matching level mapping; shell-side code logs through the same tag scheme so a single logcat filter shows the whole pipeline. Log level is a runtime setting (PRD diagnostics screen), applied at the tracing filter so disabled levels cost nothing. Player engines are wired into the same scheme: engine state transitions and EngineError emissions are logged at info/error with the channel's non-identifying id, giving support threads a coherent story across core and engine. The log-export action serializes the core's ring buffer of recent events to a shareable text file, scrubbing anything credential-shaped by construction (secrets never enter log messages because the secret types' debug representations are redacted — enforced by a custom Debug impl and a clippy-level CI grep for accidental formatting of those types; the same rule applies shell-side, where secret values never appear in interpolated log messages).

## 5. FFI boundary (UniFFI)

UniFFI in **proc-macro mode** (attributes on the Rust definitions; no separate UDL files to drift). The generated Swift package and Kotlin library are build artifacts, produced by `xtask` and consumed by the app builds — never hand-edited, never committed except as versioned release artifacts for reproducibility checks. Boundary design rules, each load-bearing: records crossing the boundary are flat, owned data (no borrowed views); every service method that touches I/O is declared async, surfacing natively as Swift async functions and Kotlin suspend functions on the core's runtime; event push uses UniFFI callback interfaces with the explicit contract that callbacks may arrive on any thread and the shells must trampoline to their main actors/dispatchers themselves; enums crossing the boundary are exhaustively matched on the shell side with an "unknown future variant" arm reserved so the core can grow without breaking older shells during development; and the boundary is versioned — a single handshake method reports core version and schema version, checked at app startup so a mismatched framework fails fast and legibly instead of corrupting data.

The threading contract in one sentence: the shells own their UI threads, the core owns all its threads, and the only crossings are async returns (delivered by the generated bindings onto the caller's continuation) and callbacks (delivered on core threads, trampolined by shell code).

## 6. tvOS application

Swift 6.3+ (whatever the current stable toolchain ships — 6.3.3 at time of writing; the docs/toolchains pin tracks the exact Xcode/Swift pair) in Swift 6 language mode with strict concurrency checking as an error, using the default-MainActor-isolation module setting so view code is main-isolated by default and anything crossing to the core is an explicit, compiler-checked boundary. UI is SwiftUI end-to-end with the Observation framework for view models; no UIKit except where tvOS focus or player surfaces require hosting. Structure is feature-modular via Swift Package Manager local packages exactly as drawn in §3.2 — CoreKit, DesignSystem, PlayerContract, PlayerMPV, PlayerAV, and one FeatureX package per vertical slice — with the app target reduced to a composition root and navigation shell, keeping build times sane and ownership per the modularity doctrine. Navigation is a state-driven stack; focus uses the native focus engine with explicit focus sections for the rail/grid/strip pattern in the PRD, and every custom focusable view implements the platform's focus appearance so the Test-Card Amber treatment rides on system behavior rather than fighting it. Images use a small async image pipeline over URLSession with a disk cache capped per the settings value; artwork is the one subsystem allowed network access outside the core, because image fetching benefits from platform caches and never touches credentials (logo URLs are public by nature; sources requiring authed artwork route through a core-provided resolver instead). Secrets live in Keychain via a thin CoreKit adapter the core calls through a host-secrets callback interface (§12). Top Shelf (P1) is an extension target reading a core-provided snapshot written to the shared app-group container. Per the Swift guidelines (`.augment/rules/swift-dev-pro.md`): package manifests declare tools-version 6.3 with the Swift 6 language mode; unit tests use **Swift Testing**, with XCTest retained only for UI tests; formatting is the toolchain-bundled swift-format with SwiftLint as the complementary linter; and **SwiftData is deliberately not used** — persistence is the core's job (§4.4), and a second source of truth on the shell would violate the architecture, so the only Apple persistence surfaces in the app are Keychain (secrets) and the image disk cache.

Playback: the engine contract (§8) has two implementations. The MPVKit engine embeds libmpv rendering into a Metal-backed hosting view, maps the contract's commands onto mpv properties and commands, forwards mpv's event stream into the contract's state machine, and configures the audio session for long-form playback (and AirPlay passthrough where formats allow). The AVPlayer engine wraps AVPlayer with the same contract for HLS-native content. tvOS-specific care points, each tracked as explicit engine acceptance tests: audio session interruption handling (Siri, phone-call handoff), the app-suspension teardown/rebuild cycle for the render surface, and honest now-playing-info reporting to the system.

## 7. Android TV application

Kotlin 2.4 (K2-only) with a multi-module Gradle setup — Kotlin DSL and a version catalog, KSP2 for annotation processing — mirroring the tvOS package layout unit for unit (§3.2): a thin single-Activity `app` composition root, `core` modules (corekit, designsystem, player-contract), peer `player` engine modules, and one `feature` module per vertical slice. UI discipline follows the Kotlin/Compose-for-TV guidelines (`.augment/rules/kotlin-dev-pro.md`) exactly: interactive components come from `androidx.tv.material3` (which bakes in focus scale/glow and D-pad semantics), never phone Material3; the removed `TvLazy*` family is never referenced — lists and grids are the foundation lazy layouts with TV pivot scrolling supplied via the bring-into-view spec; Leanback is not used anywhere; navigation is **Navigation 3**, with the back stack held as plain state; and view state exposed to composables uses immutable collections. Architecture is single Activity, ViewModel-per-feature, unidirectional state: UniFFI suspend calls are invoked from viewmodel scopes, callback events are bridged into flows by a small adapter in corekit, and cancellation propagates end-to-end (a departed screen cancels its scope, which cancels the core task through its handle). Focus behavior uses the focus-restorer modifier for restoration on back-navigation (the classic TV bug class) and explicit focus requesters for initial focus; D-pad mappings from the PRD table are implemented centrally as a key-event interceptor so behavior is uniform across screens. Dependency injection via Hilt. Images via Coil with a capped disk cache. Testing per the guidelines: kotlin.test with JUnit 5, MockK, kotlinx-coroutines-test for viewmodel logic, and Compose UI tests for focus traversal. The app targets the current API level with minSdk 26, ships as an App Bundle for Play and a fat APK for direct release, and includes the TV launcher banner, home-screen channels / watch-next integration (P1), and the system content-search provider (P1). The pairing and (future) restream servers run inside a foreground-adjacent lifecycle tied strictly to their screens per the PRD's on-while-visible rule.

Playback: the Media3/ExoPlayer (1.10.x) engine is the default — HLS, DASH, and TS through Media3's extractors, with hardware decoding, the platform media session (so the system remote and voice transport controls work), and track selection mapped onto the contract; the video surface is the Compose-native player surface from Media3's Compose artifact, not a legacy view wrapped in interop. The libmpv engine loads a prebuilt libmpv (LGPL FFmpeg configuration, arm64-v8a and armeabi-v7a device ABIs plus x86_64 for emulator) rendered into a SurfaceView, following the integration lineage of mpv-android. NDK artifacts for both the Rust core and libmpv are built per-ABI by `tools/` scripts invoked from Gradle, with checksums pinned.

## 8. The player engine contract

Both shells implement the same conceptual interface, specified here so product behavior is identical. An engine exposes: a load operation taking a stream locator, header map, user-agent, and a buffering profile; transport operations (play, pause, stop, seek-if-seekable); track enumeration and selection for audio and subtitles; aspect-ratio mode cycling; and a read-only state stream emitting a small state machine — idle, loading, buffering, playing, paused, ended, failed-with-EngineError. EngineError is a shared taxonomy with exactly the classes the PRD's error UX needs: SourceUnreachable, Unauthorized, UnsupportedFormat, DecoderFailed, Timeout, and Unknown-with-detail. Engines must be disposable and re-creatable cheaply, because zapping destroys and rebuilds them constantly; the channel strip's zap path is the performance-critical consumer and its budget (video visible within the PRD's two-second bar, UI never blocked) is an acceptance test per engine.

Engine selection policy, identical on both platforms: resolve per-channel override, else per-source override, else platform default (ExoPlayer on Android, MPVKit on tvOS). **Automatic fallback** triggers only on UnsupportedFormat or DecoderFailed from the default engine: the shell offers a one-button "Try other player" (and a remember-for-this-channel toggle), rather than silently switching — silent switching would make engine bugs invisible and support impossible. All engine parameters the settings screen exposes (buffering profile, subtitle styling) map through the contract so settings language stays engine-neutral.

## 9. Build, toolchains, and CI

Toolchains are pinned: Rust 1.96.1 via rust-toolchain file; Xcode and Android SDK/NDK versions recorded in `docs/toolchains` and asserted by build scripts. Apple targets: the core builds for aarch64-apple-tvos plus the simulator variant, packaged with generated Swift bindings into an XCFramework by `xtask`. These targets were recently promoted from Tier 3 to **Tier 2** in upstream Rust, which means prebuilt standard libraries and guaranteed-to-build status; the build documentation records the first stable toolchain shipping that promotion, and retains the previous procedure (nightly with build-std) as a documented fallback should the pin ever sit behind it. Android targets: cargo-ndk builds the three ABIs into a prefab/AAR consumed by Gradle. A reproducibility check in CI rebuilds bindings and fails on drift between Rust definitions and committed release artifacts.

CI is GitHub Actions, three lanes. The **core lane** (Linux) runs rustfmt check, clippy with the workspace lint set at deny-warnings, unit and property tests, the fixture-corpus parser tests, and cargo-deny (advisories plus the license policy in §12). The **Android lane** builds the AAR and app, runs ktlint and detekt (with the Compose ruleset), JVM unit tests, and a small Compose-for-TV instrumentation suite on an emulator including a D-pad traversal smoke test. The **Apple lane** (macOS) builds the XCFramework and app, runs swift-format and SwiftLint checks, Swift Testing unit suites, and a simulator UI smoke test. Release workflows produce signed store artifacts plus, for Android, the direct-release APK attached to a GitHub release with checksums; every release artifact embeds the exact core git revision, surfaced on the diagnostics screen.

## 10. Testing strategy

The pyramid is bottom-heavy in Rust, where the logic is. Parsers get property-based tests (random mutation of fixture playlists must never panic and must preserve the skipped-entry accounting invariant) plus a golden-fixture corpus grown from every real-world playlist bug ever reported — the corpus is the project's crown jewel and lives in `fixtures/` with provenance notes. `core-db` gets migration tests (every historical schema upgraded to head and smoke-queried) and the staging-swap refresh property (a refresh failing at any injected point leaves the prior catalog intact). The Xtream client tests against captured, scrubbed response fixtures with a local stub server. The FFI boundary gets **contract tests**: a small suite executed from both Swift and Kotlin against the real compiled core, asserting the same fixture yields the same results through both bindings — this is what keeps "parity" from being a slogan. Shell-side, view models are unit-tested against a fake CoreKit; each platform keeps a thin UI smoke suite (launch, add fixture source, reach playback screen with a stub engine) rather than a brittle full E2E net. Engine implementations get device-farm spot checks against a maintainer-operated test headend serving self-produced streams in the formats the taxonomy names — this is manual-with-checklist per release, honestly acknowledged as the least automatable layer.

## 11. Performance engineering

The PRD budgets are enforced as follows: import throughput and parser memory ceilings are asserted by criterion benchmarks in the core lane with regression gates; search latency is benchmarked at a generated 50k-channel dataset; the shells apply platform tooling per release checklist (Instruments on tvOS, Macrobenchmark and Perfetto on Android) with the low-end Chromecast-class device as the Android baseline. Structural rules that make the budgets reachable rather than aspirational: no synchronous core call on a UI thread (the FFI's async-only surface makes this nearly unrepresentable), all lists virtualized, image decode off-main, DB access through indexed query plans reviewed when query shapes change (an xtask target dumps EXPLAIN output for the hot queries), and channel-zap treated as the sacred path — profiled every release on both platforms.

## 12. Security, privacy, and license engineering

Secrets (Xtream credentials, per-source header values that may embed tokens) never persist in SQLite. The core defines a host-secrets callback interface — store, retrieve, delete by opaque key — implemented on tvOS over Keychain and on Android over Keystore-backed encrypted preferences; the DB stores only the opaque keys. Secret-bearing types in Rust redact their debug output, are zeroized on drop, and never implement serde serialization for the raw value. The pairing server binds only to LAN interfaces, exists only while its screen is visible, serves a static form and accepts one POST shape, and displays a session-random token in the TV UI that must accompany submissions — a person on the network cannot inject a source into a TV they cannot see. TLS is rustls everywhere with platform roots; the self-signed escape hatch is per-source and surfaced in the source list as a persistent badge.

License engineering: every file carries an SPDX header (AGPL-3.0-or-later for project code) checked by a REUSE lint in CI; cargo-deny enforces a dependency license allow-list (permissive plus LGPL, with copyleft-incompatible licenses denied); the mpv/FFmpeg builds on both platforms are pinned to LGPL configurations with their build flags committed, satisfying both AGPL compatibility and the App Store posture discussed in the PRD; and every screen the pairing (or future restream) server renders includes the source-code link that keeps AGPL section 13 compliance trivially true. Third-party notices are generated at build time into the About screen from the dependency graph, not maintained by hand.

## 13. Versioning and release

Semantic versioning on the app; the core carries an independent schema version (DB) and boundary version (FFI) checked in the startup handshake. DB migrations are forward-only; a downgraded app refusing a newer schema fails with a clear message rather than guessing. Release cadence follows the PRD milestones, with each release documented by a changelog generated from conventional commits. Every architecture-level decision made or changed after this document is recorded as an ADR in `docs/adr/`; this document is amended by PR like any code.

## 14. Appendix — decision log (initial ADR seeds)

**Rust core with native shells over any cross-platform UI framework.** Flutter lacks tvOS support entirely and is second-class on Android TV; Kotlin Multiplatform would share logic but forfeit the Rust engine the project explicitly wants. UniFFI gives both platforms idiomatic bindings from one definition. Consequence accepted: two UIs to maintain — mitigated by the shared engine contract and FFI contract tests.

**rusqlite over sqlx/diesel.** Embedded, in-process, single-writer SQLite behind our own async façade; the async-SQL ecosystem's costs buy nothing here, and bundling pins one SQLite version across both platforms. Consequence accepted: hand-written SQL — mitigated by migration tests and the EXPLAIN xtask.

**ExoPlayer default on Android, MPVKit default on tvOS.** On Android, Media3 is the platform-integrated path (media session, voice transport, hardware decode) and handles mainstream IPTV formats; mpv is the codec-breadth fallback. On tvOS, AVPlayer's format range is too narrow to be the default for IPTV reality, so the positions invert. Consequence accepted: different default behavior per platform — mitigated by the identical selection/fallback policy and shared error taxonomy.

**Loud fallback over silent engine switching.** Silent switching hides engine regressions and produces unexplainable behavior differences between users. The one-button fallback with remember-toggle keeps agency with the user and signal with the maintainer.

**Playback in the shells, not the core.** Media pipelines are the most platform-bound subsystem; abstracting them in Rust would maximize, not minimize, per-platform glue. The engine contract captures the sharable part — semantics — without sharing the unsharable part — implementation.

**AGPL-3.0-or-later with LGPL media libraries.** Project code is AGPL-3.0-or-later (decided) to guarantee derivative freedom including network use with future-proofing; embedded FFmpeg/mpv are built LGPL-only for App Store viability and clean compatibility. The contributor-agreement model remains the one open governance decision, flagged as launch-blocking in the PRD; recommendation on record: DCO plus explicit App Store distribution grant, decided before the first external PR is merged.

**Modularity doctrine: cohesion by principle, tooling as signal (§3.1).** An earlier draft enforced a hard ~300-line file ceiling in CI; it was rejected because mechanical size gates optimize the metric rather than the goal, pushing splits to wherever the linter fires instead of concept boundaries and trading the god-file failure mode for its confetti-code inversion. The adopted doctrine enforces structural rules absolutely (single-concept units, no junk drawers, vertical slices, downward-only dependencies, edge-only composition, earned abstraction) while running complexity/length lints at warn as review prompts with scoped, justified allowances. Consequence accepted: the standard rests on review judgment rather than automation — judged correct because cohesion is not machine-checkable, and mitigated by the newcomer test giving reviews a concrete, arguable criterion.
